## BR-CSV: CSVパースルール

### BR-CSV-01: 文字コード
- CSVはShift-JIS（CP932）で読み込む

### BR-CSV-02: ヘッダースキップ
- SBI証券CSVの先頭に存在するメタ行（「建玉一覧」等のタイトル行）をスキップし、カラムヘッダー行を検出してからデータを読み込む
- カラムヘッダー行の判定: `銘柄` または `銘柄コード` を含む行

### BR-CSV-03: 必須カラム検証
- 以下のカラムがすべて存在することを確認する:
  `銘柄（コード）`, `買付日`, `数量`, `取得単価`, `現在値`, `前日比`, `前日比（％）`, `損益`, `損益（％）`, `評価額`
- 不足カラムがある場合はパースエラーとしてインポートを中断する

### BR-CSV-04: 同一銘柄の集約
- 同一銘柄コードの複数行を1件に集約する
- **数量**: 合計（Σ 数量）
- **取得単価（加重平均）**: Σ(数量 × 取得単価) / Σ(数量)
- **評価額**: 合計（Σ 評価額）
- **損益**: 合計（Σ 損益）
- **損益率**: 集約後の 損益 / (Σ(数量 × 取得単価)) × 100
- **現在値・前日比・前日比（%）**: 同一銘柄は同値のため最初の行の値を使用

### BR-CSV-05: 数値パース
- 数値フィールドはカンマ区切りを除去してからパースする（例: `1,234,567` → `1234567`）
- `%` 記号は除去する
- `+` 符号は許可する（前日比等）

## BR-SNAPSHOT: スナップショット保存ルール

### BR-SNAP-01: 日付ユニーク・同日上書き
- 同一日付のSnapshotが存在する場合、既存のSnapshot（およびその全Holdings）を削除してから新規作成する

### BR-SNAP-02: 総計の計算
- `totalValuation` = Σ(全Holdingの totalValuation)
- `totalProfitLoss` = Σ(全Holdingの totalProfitLoss)
- `totalProfitLossPct` = totalProfitLoss / Σ(全Holdingの 数量 × 加重平均取得単価) × 100
- `holdingCount` = 集約後の銘柄数

## BR-DIFF: 差分計算ルール

### BR-DIFF-01: 比較単位
- 銘柄コードのみで同一銘柄を判定する（買付日は考慮しない）

### BR-DIFF-02: 差分分類
- **追加（added）**: 現在のSnapshotにあり、前回のSnapshotにない銘柄コード
- **除去（removed）**: 前回のSnapshotにあり、現在のSnapshotにない銘柄コード
- **変化（changed）**: 両方にあり、数量または評価額が異なる銘柄コード
- **前回なし**: 前回Snapshotが存在しない場合、差分は空（addedも表示しない）

### BR-DIFF-03: 変化の算出
- `quantityDiff` = 現在の数量 - 前回の数量
- `valuationDiff` = 現在の評価額 - 前回の評価額
- `valuationChange` = 現在の totalValuation - 前回の totalValuation

## BR-SECTOR: セクター集計ルール

### BR-SECTOR-01: 集計単位
- 東証33業種名（sector33Name）でグループ化する

### BR-SECTOR-02: 構成比
- `allocationPct` = セクター合計評価額 / 全体総評価額 × 100
- 小数点第2位を四捨五入する

### BR-SECTOR-03: メタ未取得銘柄
- J-Quantsからのメタデータ取得に失敗した銘柄は `sector33Name = "不明"` として集計する

## BR-JQUANTS: J-Quants API認証ルール

### BR-JQ-01: 認証フロー
1. 設定から `jquants.refresh.token` を取得する
2. Refresh Token → ID Token に交換する（`POST /v1/token/auth_refresh`）
3. ID Token をメモリキャッシュに保存する（有効期間: 24時間）
4. 以降のAPIコールでは ID Token をAuthorizationヘッダーに使用する

### BR-JQ-02: ID Token 再取得
- アプリ起動時、またはAPI呼び出しで401エラーが返った場合に ID Token を再取得する

### BR-JQ-03: Refresh Token 未設定時
- `jquants.refresh.token` が未設定の場合、J-Quants APIコールをスキップする
- 全銘柄の StockMeta は null として扱い、処理を継続する（グレースフルデグラデーション）

### BR-JQ-04: メタデータキャッシュ
- 取得したメタデータは `cachedAt` を記録してSQLiteに保存する
- 次回リクエスト時に `cachedAt` から24時間以内であればキャッシュを使用し、APIコールをスキップする

### BR-JQ-05: PBR/PERの取得不可対応
- 無料プランでPBR/PERが取得できない場合、フィールドは `null` とする
- プロンプト・表示ではnullフィールドを「-」として表示し、エラーとしない

## BR-GDOCS: Google Docsアーカイブルール

### BR-GDOCS-01: ドキュメントタイトル
- タイトル形式: `YYYY-MM-DD ポートフォリオスナップショット`

### BR-GDOCS-02: 同日上書き
- 指定フォルダ内にタイトルが一致するドキュメントが存在する場合、そのドキュメントを削除してから新規作成する
- （Google Docs APIの仕様上、既存ドキュメントのコンテンツ全置換より新規作成が実装しやすいため）

### BR-GDOCS-03: ドキュメント構造（リッチフォーマット）
Google Docs の構造化要素を使用する:

```
[HEADING_1] YYYY-MM-DD ポートフォリオスナップショット

[HEADING_2] サマリー
[PARAGRAPH] 総評価額: ¥X,XXX,XXX | 総損益: ±¥XXX,XXX (±X.X%)
[PARAGRAPH] 保有銘柄数: XX | スナップショット日: YYYY-MM-DD

[HEADING_2] 保有銘柄リスト
[TABLE] 銘柄コード | 企業名 | セクター | 数量 | 評価額 | 損益 | 損益率

[HEADING_2] セクター別構成比
[TABLE] セクター | 評価額 | 構成比

[HEADING_2] 前回スナップショットとの差分
[PARAGRAPH] （前回: YYYY-MM-DD）
[TABLE] 変化種別 | 銘柄 | 数量変化 | 評価額変化

[HEADING_2] AI分析プロンプト
[PARAGRAPH] （生成されたプロンプトテキスト）
```

### BR-GDOCS-04: サービスアカウント未設定時
- `google.sa.key.path` が未設定またはファイルが存在しない場合、Google Docsアーカイブをスキップする
- インポート処理自体は成功とし、レスポンスに警告メッセージを含める

## BR-ERROR: エラーハンドリングルール

### BR-ERR-01: インポート失敗（致命的）
- CSVファイルが存在しない → エラー返却、処理中断
- CSVのカラム構造不一致 → エラー返却、処理中断
- SQLite保存失敗 → エラー返却、処理中断

### BR-ERR-02: インポート継続（非致命的・警告）
- J-Quants API失敗 → メタデータなしで継続、レスポンスに `warnings` を付与
- Google Docs API失敗 → アーカイブスキップ、レスポンスに `warnings` を付与

### BR-ERR-03: エラーレスポンス
- ユーザー向けエラーメッセージは汎用メッセージのみ（スタックトレース非公開）
- 詳細エラーはサーバーサイドログにのみ記録する
